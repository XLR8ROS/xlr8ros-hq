# Navigation How-To

## Purpose
This how-to defines how XOS navigation files are created and maintained.

It exists so repo navigation stays structurally consistent as XOS grows.

## 1. Navigation purpose

Navigation files are for movement.

They answer where to go next.

They do not replace README files, registry files, or governing documents.

## 2. Structural path

The standard XOS navigation path is:

1. HQ
2. division
3. department
4. office
5. agent

Upward links should also exist where appropriate.

## 3. What gets a navigation file

Create navigation files for:

1. HQ
2. each division
3. each department when created
4. each office
5. agent repos when needed

## 4. What belongs in navigation

Navigation files should contain:

1. parent link
2. child links
3. closely related structural links
4. only the minimum descriptive context needed to move correctly

## 5. What does not belong in navigation

Do not use navigation files for:

1. full role explanations
2. full SOP text
3. long registry records
4. working notes
5. hidden private repo exposure that should stay restricted

## 6. Office and agent rule

Navigation should usually expose:

1. division links
2. office links

Agent repo links should be exposed only where appropriate for that layer.

If the office is the intended doorway, prefer the office link over the agent repo link.

## 7. Link rule

Use stable repo or file links.

When updating structure, update both:

1. downward navigation from the parent
2. upward navigation from the child

## 8. Update trigger

Update navigation when:

1. a new division is activated
2. a new department is activated
3. a new office is activated
4. an agent repo becomes part of the formal structure
5. a structural link changes

## 9. Reporting rule

When reporting navigation work, state:

1. what navigation file was updated
2. what link was added, changed, or removed
3. whether the upward and downward path is now complete
