# Agent Origination How-To

## Purpose
This how-to defines how XOS originates new agents correctly.

It exists so agents do not create primary or secondary agents outside approved authority.

## 1. Agent classes

For origination purposes, XOS recognizes:

1. primary agents
2. secondary agents
3. sub-agents

## 2. Core authority rule

Outside the designated Agent Engineering authority, agents may not originate new primary or secondary agents.

Outside the designated Agent Engineering authority, agents may originate only sub-agents within authorized scope.

## 3. Temporary authority rule

Until a dedicated Agent Engineering authority exists, primary and secondary origination authority remains with the currently designated supervising authority.

For SEAD-side origination, that temporary authority currently sits with the Chief of SEAD.

## 4. When a new primary or secondary agent is requested

When a request appears for a new primary or secondary agent:

1. identify the requested seat
2. identify the parent structure
3. confirm whether the requester has origination authority
4. if not, route the request to the proper authority
5. do not silently create the new agent anyway

## 5. Minimum origination package

Before a new primary or secondary agent is treated as active, define at minimum:

1. name
2. role or seat
3. parent division or department
4. office placement
5. reporting line
6. repo decision
7. communication identity
8. governing documents
9. navigation placement
10. activation status

## 6. Sub-agent origination procedure

A sub-agent may be originated only when:

1. the work is narrow
2. the work is delegated
3. the work stays inside the parent authority boundary
4. the sub-agent is not being used to bypass approval or structure

When spawning a sub-agent, define:

1. objective
2. scope
3. inputs
4. expected output
5. stop condition
6. escalation path

## 7. Activation rule

A proposed agent may exist in planned state before activation.

Do not treat a planned agent as active until its minimum operating package is materially in place.

## 8. Escalation rule

Escalate when:

1. the requested agent class is unclear
2. the parent structure is unclear
3. authority is unclear
4. the request would materially change structure or spend
5. the request appears to bypass Agent Engineering authority

## 9. Reporting rule

When reporting agent origination work, state:

1. what class of agent is being requested
2. who holds origination authority
3. whether the request is approved, denied, planned, or routed
4. what minimum package remains missing
