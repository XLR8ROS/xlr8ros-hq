# Registry How-To

## Purpose
This how-to defines how XOS registries are structured and maintained.

It exists so structural identity, reporting lines, and communication surfaces remain clear.

## 1. Registry purpose

A registry records what exists.

A registry is not the same thing as navigation.

Registry answers:

1. who exists
2. what seat they hold
3. where they sit
4. how they are reached
5. who they report to
6. what status they are in

## 2. Current registry format

Until otherwise changed, the recommended live registry format is a structured table or sheet.

CSV exports may be used for snapshots.

## 3. Core columns

Use these columns in this order:

1. Agent
2. Email
3. Position
4. Division
5. Department
6. Office
7. Supervisor
8. Supervisor Position
9. Supervisor Email
10. Status
11. Notes

## 4. Linking rule

Use links for structural destinations such as:

1. division
2. department
3. office

Do not expose agent repo links in the shared registry unless that exposure is intentionally approved.

## 5. Office rule

Office is the primary structural doorway.

If a seat has an office, the registry should generally point to the office rather than to the private working repo of the agent.

## 6. Status rule

At minimum, status should distinguish:

1. planned
2. active
3. dormant
4. retired

## 7. Update trigger

Update the registry when:

1. a new agent is activated
2. a position changes
3. a reporting line changes
4. an office changes
5. a communication identity changes
6. a unit changes status

## 8. Reporting rule

When reporting registry work, state:

1. what record changed
2. what fields changed
3. whether the structure now matches the current operating reality
