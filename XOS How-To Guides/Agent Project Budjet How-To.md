# AGENT PROJECT BUDGET HOW-TO

## Timestamp
20260411 114900 EDT

## Purpose
This how-to defines how to prepare a one-time agent project budget.

It is for bounded work with a defined end state.

It is not for ongoing recurring work.

## Core rule
This budget should stay simple.

It does not need to become a full project proposal.

It only needs to clearly show:

1. what success looks like
2. what tasks make up the work
3. how many sub-agents are needed for each task
4. how much time each task should take
5. how many tokens each task is expected to use
6. how much each task is expected to cost
7. what the total time and total cost are

## Pricing rule
Unless instructed otherwise, use the most expensive applicable API agent as the pricing assumption.

This keeps the estimate conservative by default.

## Structure

### Part A

#### A1. Title
State the name of the work being budgeted.

#### A2. Description
State what the work is and what success looks like.

Keep this plain and direct.

## Part B

Part B is task-by-task.

Each task gets its own B-number.

### B1. Task 1
State the first task.

#### 1.1 Dependency
State what must be completed before this task starts.

If nothing must be completed first, state that clearly.

#### 1.2 Number of sub-agents
State how many sub-agents are needed for this task.

#### 1.3 Estimated time
State how long this task should take.

#### 1.4 Input tokens
State the expected input tokens for this task.

#### 1.5 Input token cost
State the estimated cost of the input tokens for this task.

#### 1.6 Output tokens
State the expected output tokens for this task.

#### 1.7 Output token cost
State the estimated cost of the output tokens for this task.

#### 1.8 Total task cost
State the total estimated cost for this task.

### B2. Task 2
State the second task.

#### 2.1 Dependency
State what must be completed before this task starts.

#### 2.2 Number of sub-agents
State how many sub-agents are needed for this task.

#### 2.3 Estimated time
State how long this task should take.

#### 2.4 Input tokens
State the expected input tokens for this task.

#### 2.5 Input token cost
State the estimated cost of the input tokens for this task.

#### 2.6 Output tokens
State the expected output tokens for this task.

#### 2.7 Output token cost
State the estimated cost of the output tokens for this task.

#### 2.8 Total task cost
State the total estimated cost for this task.

### B3. Task 3
Continue the same structure for each additional task until the work is fully budgeted.

## Part C

### C1. Total input tokens
State the total input tokens across all tasks.

### C2. Total input cost
State the total input token cost across all tasks.

### C3. Total output tokens
State the total output tokens across all tasks.

### C4. Total output cost
State the total output token cost across all tasks.

### C5. Total time
State the total estimated time across all tasks.

### C6. Overall total sub-agent cost
State the full estimated sub-agent cost for the entire one-time project.

## Working standard
Use clear answers, not filler.

If the task chain is simple, keep it short.

If the task chain is complex, break it down cleanly.

Do not add extra bureaucracy unless explicitly requested.

## One-time project rule
Use this how-to only when the work is bounded and has a clear finish.

If the work is recurring or ongoing, use the ongoing agent budget format instead.

## What changes for ongoing budgets
The ongoing version should keep the same logic, but it should be budgeted by time period instead of end-state.

Example periods include:

1. weekly
2. monthly
3. quarterly

The one-time version asks:
what does success look like, and what does it take to get there?

The ongoing version asks:
what does it cost to keep this running during the selected period?
